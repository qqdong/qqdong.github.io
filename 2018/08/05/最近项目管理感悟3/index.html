最近项目管理感悟3

发表于 2018-08-05 | 分类于 summary

最近2个月，间断地写了几篇管理、沟通相关的文章，技术文章没在写了，更新速度也远没有年初几个月那么快，一方面自己慢慢向管理转型，一些流程和规范要慢慢梳理和适应，另一方面就是自己有点懒了，勤快点还是有时间学习和总结技术知识的。

自己的内心和朋友都提醒我，技术不能放下，要一直学习、实践、总结、分享下去。后面我们组有2个重点专项突击项目，我会重点跟进数据清洗和数据分析相关的改进优化，要好好学习一段时间了。

管理上，最近1个月也经历了一些事情，从大佬的言行中有一些体会和感悟，从几个方面总结下。之前写过2篇感悟，感兴趣的朋友可以一起讨论：

  • 管理感悟篇1
  • 管理感悟篇2

做事情的成本和价值意识

前几天参加了一个讨论会，主要议题是后续的2个大需求。一个需求是要实现一个通用的客户端SDK，可以快速接入我们的客服系统，这个需求是领导提出来的，因为要接入客服系统的业务方会越来越多，提供一个稳定、通用的SDK可以减少接入成本和后续的开发成本。

另一个需求是，有品想开发一个商家APP，集成客服工作台、工单、各种查询工具等功能，因为只是沟通过程中的一个想法，没有深刻理解和分析，讨论过程中，认识到领导很关注投入产出比，以及后续的价值。

可以看到，领导很重视做一件事情的成本和价值，如果成本比产生的价值高，做的必要性就不大，除非有别人为你买单或是必须要做的事情，这是判断一件事情要不要做很重要的思路。

一个需求过来，最终把它实现的是团队的兄弟们，如果可以让他们意识到做事情的价值，会更用心、更多思考、更有劲头地去做，而不仅仅是简单的实现。

开发质量意识

我们的客服系统是7*24小时服务的，线上的问题可能会影响很多用户，如果问题比较多，会引起不少投诉，有损公司的形象，丧失一部分用户，这是很严重的事情。

大家都在网上买过东西，收到东西后如果不满意，可能会通过一些途径联系客服，我们系统提供在线、热线、邮件的渠道，如果在聊天的过程中出现丢消息、会话异常的问题，会很影响自己的心情。

在开发时，如果有这种意识，就会很注重代码的质量，主动考虑可能发生的异常情况，让系统更稳定，很多人做不到这一点，我做的也不够好。

如果系统有大的功能变更或优化改造，需要在上线前check下关键场景，在测试环境下验证好。

对上汇报，要深思熟虑

工作过程中会产生很多想法，包括自己对一些问题的思考，或者和业务方沟通过程中，了解到他们的一些想法，有时我会比较着急，急着去实施或者向领导汇报，但效果不是特别好，这是我以后要注意的。

一个想法从诞生到实施需要做很多事情，整合、梳理自己的想法，和同事或有经验的人交流意见，给出一个初步的实施方案，开会确认方案的可行性和完整性。领导关心的是实施方案，而不仅仅是一个想法，不要急于汇报自己没想清楚的事情。

对上汇报的事情，是要对其负责的，不是说说那么简单，如果领导只听到你的「雄心」、你的规划，而迟迟没有成果，或者你经常夸大一些事情，慢慢地会失去领导的信任，得不到重用。

对上汇报，要深思熟虑，说的话要有意义和价值。

加强沟通和协作

部门之间有座大墙，小组之间有座小墙，人与人之间有个鸿沟。

要把一件事情做好，绝不是一个人的事情，需要团队协作，小组、部门相互合作，平时主动了解他们在做什么，使用了哪些好的技术方案，需要的时候好找他们合作。

开发也只是其中一部分，视野要广阔些，不要只看着自己手上的一摊事，多和需求涉及方沟通，发现真正的问题，才能够更好的解决。

执行过程中，以「把事情做好」为最终目的，对事不对人，推进一些事情的时候，多沟通，不要有太多顾虑，不要担心会不会打扰别人时间、一些问题会不会很菜等。

细节决定成败

在一次给部门大Boos汇报的会议上，明显感觉到他很注重细节，特别是用户体验相关的。

想想也对，如果在细节上做的都很好，整体上也不会差，这是一种态度。

做事情时，不要忽略每一个细节，特别是和直接用户相关的。
