部门异地办公研讨会

发表于 2018-06-10 | 分类于 summary

昨天，部门组织了异地办公研讨会，很高兴能够参与其中，大佬们的倾心分享、过程中的小游戏、长时间的分组讨论、会后的小聚餐，每个环节都有所感悟，值得在以后的工作中重视起来，不断运用。

为了减少人力成本，部门在武汉成立了研发中心，正有计划地把一些项目和新需求迁移到武汉，研发人员分隔两地，如何更高效地「异地办公」就是本次的主要议题。

会议进行了一整天，上午主要介绍了武汉那边的情况，包括员工、资源、在做的一些项目需求，各个业务组负责人也分享了自己的思考和大致计划。下午针对一些关键问题分组进行讨论，每个组选一个议题，讨论完成后，小组组长向所有人阐述小组的讨论成果。

下面的内容很多与「异地办公」无关，只是大佬言语中给我的感悟，不断学习。

异地办公问题

我总结主要有3个问题：人的问题、项目迁移问题、沟通协作问题。

人的问题，包括武汉人才招聘、人才培养、公司文化的传承，针对这块，HR雅静进行了详细介绍，明确的社招、校招计划，清晰的培养和激励计划，公司「文工团」的巡演等。

项目迁移问题，主要是迁移范围和迁移方法，各个业务负责人分别进行了详细介绍，关键点主要是新人快速熟悉系统和线上问题及时处理。

沟通协作问题，是异地办公最大的问题，因为距离原因，沟通效率会很低，如果对需求理解不一致会造成严重后果。主要思路是如何减少沟通，通过明确、详细的产品需求文档，完善的开发流程和平台化，可以大大减少不必要的沟通。

两地同心，高效协作。

传词游戏

下午的开场，做了一个小游戏「传词游戏」，应该好多人知道游戏规则：几个人一组，第1个人会被告知一个词或成语，仅使用动作描述这个词或成语，依次传递下去，最后一个人猜。

这个游戏和讨论主题很契合，一个需求或问题，经过多人传递后，会导致理解偏差，应减少沟通的传递次数和频率，尽量用标准的规范和流程去约定它。

每个参与者都应该清晰、明确传递者要表达的内容并向下正确传递，一个人有问题，最后猜到结果就会很困难，流程可以规范和强化这个过程，文档是流程和沟通的载体。

关注自己的进步

说到这个，是由2个事情引起的。

第一个事，我们小组的组长是按年龄选出来的，年龄最小的作为组长，组长阐述环节，年轻的丁一同学已经表达的很不错了，但他自己一直说表现的不好，只想说：第一次已经很不错了，换做是我，我会特别紧张，还没你说的好，相信下一次你会进步很多。

第二个事，就是晚宴环节，我从内心不太适应这种场合，害怕冷场，又不太主动和别人聊一些事情，换做之前的我，还会责怪自己，现在的我虽说还是不善言谈，但已经坦然接受了，这就是进步吧，相信随着不断观察和学习，会不断改善的。

不要拿自己的不足和别人的长处比，多关注自己的进步，有了这个心态，慢慢的会好起来。

仪式感

大领导的参与、红艳的横幅、会后的晚宴，都彰显了对本次讨论会的重视，这样一来，每个参与者也会很重视，觉得这是公司重要的事，要全力配合和支持。

这种仪式感很重要，每周的周会可以让大家自觉的梳理本周做的事情和下周计划，项目立项会可以让业务、产品、研发、测试对相关节点达成一致，不能随心所欲，拖拖延延。

仪式感可以让人对某件事情重视起来，从而有效的执行。

靠近主航道

公司都是有核心战略的，一个部门也是，有自己的主航道，主航道明确了，各个业务小组只有往上靠，偏离了，整个方向就错了。

还有一点，你的上级领导只会告诉你方向和目标，具体怎么做是你要想的，这是你的职责，你反映给领导的不是一堆抱怨和问题，而是你的解决方案和具体实施计划。

不要固守自己的一亩三分地和日常任务，多做一些事情，往主航线靠近、前行，才会有进步，得到领导的认可。

个人影响力

影响力是一个人很重要的能力，特别是领导，跟个人为人处事和做事风格关系很大，影响力高可以很容易去推动一个事情。

对内要关注每个团队成员的成长，让他们做自己想做的事情，尽力帮他们解决棘手的问题，为他们争取福利。

对外要多沟通交流，合作时要体现团队的专业力，做出来的东西让他们满意，响应速度要快，质量要有保障。

多向大佬们学习。

酒

酒就不多说了，促进大家的感情，相互了解，也是为了工作中更好的沟通和协作，喝多喝少无关紧要，重要的是交流氛围。

为什么会出现HBase

任何一个技术的出现都是有原因的，了解它为什么出现，以及它解决了什么问题，更有助于理解它的特性和设计思想。

MySQL瓶颈

MySQL是一个关系型数据库，有很高的数据一致性和持久性保证，当访问量特别高时，特别是写入操作，会有很大的O性能瓶颈。

虽然可以通过主从读写分离、分库分表的方式解决，但随着数据量不断增大、并发不断增高，MySQL应用开发越来越复杂，也越来越具有技术挑战性。

另外，分表分库的规则的设定都是需要经验的，虽然有Cobar、MyCat、Sharding-JDBC、TDDL、DBProxy中间件层来屏蔽开发者的复杂性，但是避免不了整个架构的复杂性。

分库分表的子库到一定阶段又面临扩展问题，需求的变更可能又需要一种新的sharding。

MySQL的扩展性差、大数据下IO压力大、表结构更改困难，正是MySQL开发人员面临的问题，也是MySQL的瓶颈。

虽然有这些瓶颈，但对数据一致性要求特别高的业务，还是要使用它，但是有的应用场景不需要太高的一致性，在大数据量、高并发的业务中，可以选择其他存储方案。

常见优化方法

联合索引最左前缀原则

复合索引遵守「最左前缀」原则，查询条件中，使用了复合索引前面的字段，索引才会被使用，如果不是按照索引的最左列开始查找，则无法使用索引。

比如在(a,b,c)三个字段上建立联合索引，那么它能够加快a|(a,b)|(a,b,c)三组查询的速度，而不能加快b|(b,a)这种查询顺序。

另外，建联合索引的时候，区分度最高的字段在最左边。

为什么需要索引

总体来说，索引是为了提高查询速度的，当数据量比较大时，从头到尾依次检索是无法接受的。另外，存储的数据会包含多个属性来描述业务实体，属性可以连续或分开存储，分别对应到MySQL和HBase。

以MySQL为例，学生这个实体有学号、姓名、性别、所属班级等属性，一般还有个主键ID，现在有个需求：查询学号为002的学生姓名。

数据是存储在磁盘上的，操作系统读取磁盘的最小单位是块，如果没有索引，会加载所有的数据到内存，依次进行检索，加载的总数据会很多，磁盘IO多。

如果有了索引，会以学号为key创建索引，MySQL采用B+树结构存储，一方面加载的数据只有学号和主键ID，另一方便采用了多叉平衡树，定位到指定学号会很快，根据关联的ID可以快速定位到对应行的数据，所以检索的速度会很快，因为加载的总数据很少，磁盘IO少。

可见，索引可以大大减少检索数据的范围、减少磁盘IO，使查询速度很快，因为磁盘IO是很慢的，是由它的硬件结构决定的。

编计划

编写计划时，需不要团队成员参与，如果是你，你会采用下面哪种方式？
A：你写出项目的计划并发给成员，请他们了解并有效执行；
B：让团队成员分别计划，你再最终整合；

可以通过分析下面的情况作出最后的选择：

  • 管理者的成熟程度：如果有很多这方面的经验，有成功的解决方案，可以偏向A；
  • 执行者的成熟程度：如果任务最终执行者这方面的经验比较丰富，可以偏向B；
  • 计划的紧迫程度：如果很紧迫，没有太多时间让每个成员都参与进来，可以偏向A；
  • 成员的参与程度：如果想提高成员的参与感，可以偏向B；

确定计划或里程碑后，需要进行细化，对工作进行分解，按照目标、活动、任务的结构进行细分，可以通过常用的思维导图工具进行分析。

进行任务分解时，要考虑全面，注意下面几类工作：

  • 技术 + 管理类
  • 自己的 + 相关方的协调和对接
  • 单元类 + 整合类
  • 实现类 + 转移类，转移给其他人继续进行

教授方式

培训室放了6个长桌，相应的分成6组，每组7个人，围在一起，方便讨论。

开始时，让每个人在一张A4纸描出左手的形状，从大拇指到小拇指分别填写自己的家乡、爱好、工作岗位、星座、最想去的城市，很快大家都相互熟悉了。

每个组会选出一个组长，通过刚刚「五指破冰」的相互了解，每个人任一指定一个人作为理想组长，被指最多的人会成为组长。

培训内容覆盖项目管理全过程，以「过游戏关卡」的方式组织内容，逐一讲解，打怪升级。

培训过程中，老师会抛出问题，小组讨论后派一个代表作答，根据回答的质量或抽扑克算相应的得分，有助于小组间良性竞争，提高积极性。

为了提高实践性，每组选一个自己做过的项目，辅以“一页纸项目管理”工具，全程讨论和演练。

因为时间的原因，后面的几个过程稍稍带过，略表遗憾。
